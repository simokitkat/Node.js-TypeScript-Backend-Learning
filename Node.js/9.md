# Lesson 09: Loading Local Modules with `require()`

## 🧠 Mental Model: The Delivery System

`require()` is like a delivery request. You hand Node a path (`./add`), it fetches the file, **executes its entire contents synchronously**, packages up what's been exposed, and returns it. The file runs in a sealed container—nothing leaks into your main app unless you explicitly ask for it.

## 📦 Core Concepts

| Concept                 | Explanation                                                                                                |
| ----------------------- | ---------------------------------------------------------------------------------------------------------- |
| **File Isolation**      | Every `.js` file runs in its own scope. Code in `add.js` is invisible to `index.js` until loaded.          |
| **CommonJS Standard**   | Node.js adopted CommonJS (`require` / `module.exports`) as its default module system.                      |
| **Path Resolution**     | `./` = relative to current file. Bare names (`fs`) = built-in. Package names (`express`) = `node_modules`. |
| **Extension Fallback**  | If no extension is provided, Node tries: `.js` → `.json` → `.node` in order.                               |
| **Synchronous Loading** | `require()` blocks execution until the file is read, parsed, and executed.                                 |

## 💻 Minimal Working Example

```js
// add.js
const add = (a, b) => a + b;
const result = add(1, 2);
console.log("Module executed:", result); // Runs first

// index.js
console.log("Before require");
require("./add"); // .js extension is optional
console.log("After require");
```

**Terminal Output:**

```
Before require
Module executed: 3
After require
```

## 🔄 Execution Flow

```
1. Node starts index.js
2. Hits require('./add')
3. ⏸ Pauses index.js → loads add.js
4. ▶ Executes add.js top-to-bottom → logs "3"
5. ✅ Returns control to index.js
6. ▶ Continues remaining lines → logs "After require"
```

> 💡 **Debugging Insight**: In VS Code, step-through (`F11`) shows execution physically jumping into the required file, running its code, then returning to the original call stack.

## 🛡️ Production-Ready Patterns

```js
// ✅ ALWAYS use relative paths for local modules
require("./utils/math"); // Relative to current file
require("../models/user"); // Parent directory

// ✅ Omit .js for cleaner imports (Node infers it)
require("./config"); // Node checks config.js automatically

// ✅ Keep requires at the top (synchronous by nature)
const config = require("./config");
const helpers = require("./utils/helpers");

// ⚠️ NEVER omit ./ or ../ for local files
require("add"); // ❌ Tries node_modules/add or built-in 'add'
require("./add"); // ✅ Correct
```

## 🧠 Interview Prep Hook

> **Q: "What happens under the hood when `require('./module')` is called?"**  
> **A**: "Node resolves the path, checks the internal module cache. If uncached, it reads the file, wraps it in a CommonJS IIFE `(function(exports, require, module, __filename, __dirname) { ... })`, executes it synchronously, caches the `module.exports` object, and returns it. Subsequent calls return the cached object without re-execution."

> **Q: "Why does Node automatically resolve missing `.js` extensions?"**  
> **A**: "Node's resolver maintains a fallback array: `['.js', '.json', '.node']`. When an extension is omitted, it iterates through this list, attempting to resolve and load the first match. This is a developer convenience, but explicit extensions can improve IDE intellisense and build tool compatibility."

> **Q: "Why are local modules isolated by default?"**  
> **A**: "Each file is executed in a private scope via a module wrapper function. This prevents global namespace pollution, eliminates accidental variable overwrites, and enforces explicit dependency declaration—critical for maintainability in large applications."
