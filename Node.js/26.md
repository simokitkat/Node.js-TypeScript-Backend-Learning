# Lesson 26: The `fs` Module — File System Operations

## 🧠 Mental Model: The File System Bridge

The `fs` (File System) module is Node.js's interface to your computer's storage. It offers two execution modes for every operation: **Synchronous** (blocks the main thread until completion) and **Asynchronous** (starts the task, frees the thread to run other code, and calls a function when done). Async is the recommended default for server performance.

## 📦 Import & Setup

```js
const fs = require("node:fs");
```

> ✅ The `fs` module exposes methods to read, write, and modify files on disk.

## 📖 Reading Files

### Synchronous: `readFileSync()`

```js
// Blocks execution until file is fully read
const fileContents = fs.readFileSync("./file.txt", "utf-8");
console.log(fileContents); // 'hello code evolution'
```

- Returns data directly.
- Requires `'utf-8'` encoding to get a human-readable string. Without it, returns a raw `Buffer` (binary data).
- ⚠️ **Blocks the main thread**. Acceptable only for critical startup tasks (e.g., reading config), but poor for large files or concurrent users.

### Asynchronous: `readFile()`

```js
// Non-blocking: execution continues immediately
fs.readFile("./file.txt", "utf-8", (err, data) => {
  if (err) {
    console.log(err);
    return;
  }
  console.log(data); // 'hello code evolution'
});
console.log("This runs BEFORE the file is finished reading");
```

- Keeps track of the callback and allows subsequent code to execute.
- When the OS finishes reading the file, Node.js executes the callback.
- Prevents app freezing under concurrent load.

## ✍️ Writing Files

### Synchronous: `writeFileSync()`

```js
fs.writeFileSync("./greet.txt", "hello world");
// Creates file if it doesn't exist
// Overwrites existing content if it does
```

### Asynchronous: `writeFile()`

```js
// Overwrite (default)
fs.writeFile("./greet.txt", "hello vishwas", (err) => {
  if (err) console.log(err);
  else console.log("File written");
});

// Append to existing content
fs.writeFile("./greet.txt", " hello vishwas", { flag: "a" }, (err) => {
  if (err) console.log(err);
  else console.log("Content appended");
});
```

- Accepts an options object as the third argument.
- `{ flag: 'a' }` preserves existing content and adds new text to the end.

## 📋 The Error-First Callback Pattern

Node.js standardizes asynchronous callbacks using the **error-first pattern**:

```js
(err, data) => {
  if (err) {
    /* Handle error */
  } else {
    /* Process data */
  }
};
```

- **`err`**: Contains error details if the operation fails, or `null` if successful.
- **`data`**: Contains the result/file contents only when `err` is `null`.
- This pattern is used consistently across Node.js core modules.

## ⏱️ Proving Non-Blocking Behavior

```js
console.log("first");

fs.readFileSync("./file.txt", "utf-8"); // Blocks here

console.log("second");

fs.readFile("./file.txt", "utf-8", (err, data) => {
  console.log("fourth: async callback runs after file is ready");
});

console.log("third");

// Output order: first → second → third → fourth
```

> 💡 The async callback runs last, proving that Node.js does not wait for I/O operations to complete before moving to the next line.

## 💡 Key Takeaway

The `fs` module provides both synchronous (`*Sync`) and asynchronous (`*`) methods for file operations. Synchronous methods block the single-threaded event loop and should be avoided in production servers. Asynchronous methods use the error-first callback pattern to keep the thread free, enabling Node.js to handle multiple concurrent users efficiently without freezing. The `fs` module internally uses buffers for data transfer, which is why specifying encoding (`utf-8`) is necessary for readable strings.

## 🧠 Interview Prep Hook

> **Q: "What's the difference between `readFileSync` and `readFile` in Node.js?"**  
> **A**: "`readFileSync` is synchronous and blocks the main thread until the entire file is read, returning the data directly. `readFile` is asynchronous; it initiates the read operation, continues executing subsequent code, and invokes an error-first callback once the file is ready. Async is preferred for servers to avoid blocking concurrent requests."

> **Q: "What is the error-first callback pattern in Node.js?"**  
> **A**: "It's a standard convention for asynchronous callbacks where the first parameter is always `err` (null on success, error object on failure) and subsequent parameters contain the result data. Developers check `if (err)` first to handle failures before processing the returned data."

> **Q: "Why does `fs.readFileSync('./file.txt')` return a Buffer instead of a string?"**  
> **A**: "The `fs` module internally uses buffers to handle raw binary data from the file system. Without specifying an encoding like `'utf-8'`, Node.js returns the raw buffer. Passing `'utf-8'` as the second argument tells Node to decode the buffer into a human-readable string."

> **Q: "How do you append to a file instead of overwriting it in Node.js?"**  
> **A**: "Pass an options object with `{ flag: 'append' }` to `fs.writeFile()` or `fs.writeFileSync()`. This instructs the file system to add new content to the end of the existing file rather than replacing it."

> **Q: "Why is synchronous file I/O discouraged in a Node.js web server?"**  
> **A**: "Node.js runs on a single thread. Synchronous I/O blocks that thread until the operation completes. During that time, the server cannot handle any other incoming requests, leading to poor performance, increased latency, and frozen behavior under concurrent load."
