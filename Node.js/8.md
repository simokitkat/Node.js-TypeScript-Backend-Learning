# Lesson 08: Node.js Modules — Foundation & Types

## 🧠 Mental Model: The Sealed Container

Think of every `.js` file as a **sealed container**. Code inside runs in its own private space. Nothing leaks out unless explicitly exposed. Nothing comes in unless explicitly requested. This prevents global namespace collisions and enables scalable code architecture.

## 📦 What Is a Module?

A module is an **encapsulated, reusable chunk of code** with its own execution context.

**Node.js Rule:** `1 file = 1 module`

- Every `.js` file you write is automatically wrapped in a module function by Node.
- Variables, functions, and constants defined inside are **private to that file**.

## 📊 The 3 Module Types in Node.js

| Type                | Origin                       | Location                       | Example                                  |
| ------------------- | ---------------------------- | ------------------------------ | ---------------------------------------- |
| **Local / Custom**  | Written by you for your app  | Project directory              | `./utils/math.js`, `./models/user.js`    |
| **Built-in / Core** | Shipped with Node.js runtime | Node.js internals              | `fs`, `path`, `http`, `crypto`, `events` |
| **Third-Party**     | Written by the community     | `node_modules/` (via npm/yarn) | `express`, `lodash`, `axios`, `dotenv`   |

> 🔍 **Current Section Focus**: Local modules (how to structure, expose, and consume your own code). Built-in and third-party covered in Sections 3 & 5.

## 🔒 Core Principle: File-Level Scope Isolation

```js
// moduleA.js
const secret = "hidden";
function calculate() {
  return 42;
}

// moduleB.js
console.log(secret); // ReferenceError: secret is not defined
```

> ✅ **Why this matters**: Node.js does **not** use global scope by default. Every file gets its own lexical scope. You must explicitly share code via the module system.

## 🛠️ Production Implication

- **Predictability**: No accidental variable overwrites across files.
- **Maintainability**: Clear dependency graphs → easier testing, refactoring, and code reviews.
- **Security**: Sensitive logic/data stays isolated unless intentionally exported.

## 🧠 Interview Prep Hook

> **Q: "How does Node.js handle module scoping?"**  
> **A**: "Every file in Node.js is treated as an independent module with its own execution context. Variables defined in one file are not accessible in another unless explicitly exported. This module-level isolation prevents global namespace pollution and enforces explicit dependency declaration."

> **Q: "What are the three categories of modules in Node.js?"**  
> **A**: "Local modules (developer-created, file-based), Built-in/Core modules (shipped with the runtime like `fs` or `http`), and Third-party modules (community packages installed via npm)."

> **Q: "Why does Node.js enforce file-level module scope instead of global scope?"**  
> **A**: "Global scope leads to naming collisions, tight coupling, and unpredictable side effects in large applications. Module scoping enforces encapsulation, makes dependencies explicit, and aligns with modern software engineering principles like separation of concerns."
