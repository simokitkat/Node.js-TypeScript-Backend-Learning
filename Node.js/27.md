# Lesson 27: The `fs/promises` Module

## 🧠 Mental Model: The Modern Async Wrapper

The callback-based `fs` API works, but nesting callbacks can become hard to read. The `fs/promises` module wraps the same file system operations in **Promises**, enabling cleaner asynchronous syntax (`.then()/.catch()` and `async/await`) while maintaining the same non-blocking behavior.

## 📦 Import & Setup

```js
const fs = require("node:fs/promises");
```

> ✅ Append `/promises` to the standard `node:fs` path to access the promise-based API.

## 🔗 Promise Syntax (`.then()` / `.catch()`)

```js
fs.readFile("file.txt", "utf-8")
  .then((data) => console.log(data))
  .catch((err) => console.error(err));
```

- `.then()` executes when the file read completes successfully.
- `.catch()` executes if the operation fails (e.g., file not found, permission denied).

## ⏳ Async/Await Syntax (`try` / `catch`)

Top-level `await` requires `.mjs` files or `"type": "module"` in `package.json`. In standard `.js` files, wrap the logic in an `async` function:

```js
async function readFile() {
  try {
    const data = await fs.readFile("file.txt", "utf-8");
    console.log(data);
  } catch (err) {
    console.error(err);
  }
}

readFile();
```

> ✅ `async/await` is syntactical sugar over Promises. It makes asynchronous code read like synchronous code, improving readability.

## ⚡ Proving Non-Blocking Behavior

```js
console.log("first");

fs.readFile("file.txt", "utf-8").then((data) => console.log(data));

console.log("second");

// Terminal Output:
// first
// second
// [file contents]
```

Node starts the file read, sets it aside, continues executing the next line (`console.log('second')`), and only invokes `.then()` when the OS signals the read is complete. This prevents the main thread from freezing under concurrent load.

## ⚖️ Performance Consideration

- **Callback-based `fs` APIs** are preferable when **maximal performance** is required (both execution time and memory allocation).
- If performance is not a critical concern, the **promise-based `fs` module** is recommended for improved code readability and maintainability.

## 💡 Key Takeaway

The `fs/promises` module provides a modern, promise-driven interface for file system operations. It supports both `.then()/.catch()` chains and `async/await` syntax, making asynchronous code easier to read and debug. While callbacks offer slight performance advantages for high-throughput scenarios, promises are the preferred choice for most applications due to cleaner control flow.

## 🧠 Interview Prep Hook

> **Q: "How do you use the promise-based `fs` module in Node.js?"**  
> **A**: "Import it via `require('node:fs/promises')`. It returns Promises from methods like `readFile()` and `writeFile()`, allowing you to use `.then()/.catch()` or `async/await` instead of error-first callbacks."

> **Q: "Why does the video wrap `async/await` in a function instead of using it at the top level?"**  
> **A**: "Top-level `await` is only supported in ES modules (`.mjs` files or when `"type": "module"` is set in `package.json`). In standard CommonJS `.js` files, `await` must be used inside an `async` function."

> **Q: "When should you prefer callback-based `fs` APIs over promise-based ones?"**  
> **A**: "Callback-based APIs are preferable when maximal performance is required, specifically regarding execution time and memory allocation. Promises introduce slight overhead, so they're best used when performance isn't the primary constraint and code readability is prioritized."

> **Q: "Does `fs/promises` change the underlying asynchronous behavior of file I/O?"**  
> **A**: "No. It's a syntactical wrapper over the same non-blocking I/O operations. The main thread remains unblocked, and file operations still execute asynchronously via the event loop."
